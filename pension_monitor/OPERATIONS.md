# 운영 가이드 — 토큰/비용 최소화

## 핵심 원칙
주간 파이프라인은 **GitHub Actions에서 순수 Python으로 실행**되어 LLM 토큰을
전혀 쓰지 않는다. 토큰은 오직 "사람/Claude가 실행 결과를 들여다볼 때"만 든다.
따라서 **Claude를 정상 루프에서 빼는 것**이 비용 최소화의 핵심이다.

## 상태별 동작

### 1. 정기 (매주 월 09:00 KST) — 토큰 0
- 스케줄 cron이 기본 브랜치에서 자동 실행. 수집→DB 동기화→리포트→메일까지 무인.
- Claude 개입 불필요. 결과는 메일 + `reports/latest.md` + DB로 확인.

### 2. 별도 요청 시 (온디맨드) — 토큰 최소
사용자가 "업데이트 해줘"라고 하면 Claude는:
1. `workflow_dispatch`로 `pension-events` 워크플로 트리거
2. 완료 대기 후 **작은 파일 2개만** 읽는다 (절대 잡 로그 전체를 끌어오지 않음):
   - `data/last_run_summary.json` — 건수/실패/신규·종료 요약 (~300B)
   - `reports/latest.md` — 직전 리포트 전문 (작음)
3. 위 요약으로 사용자에게 보고

### 3. 실패 시에만
- 먼저 **GitHub Step Summary 한 줄**을 확인 (로그 전체 불필요).
- 그래도 원인 불명일 때만 해당 잡의 로그를 `tail_lines` 작게 조회.

## 관측 비용을 키우는 안티패턴 (하지 말 것)
- `actions_list`로 워크플로 실행 목록 조회 → 응답에 저장소 JSON이 통째로
  중복돼 100KB+. 완료 감지는 `git fetch` 후 브랜치 HEAD 커밋 변화로 판단.
- `get_job_logs`로 풀 로그 조회 → 정상 케이스에선 불필요. 요약 파일로 대체.
- `events_latest.json` 전건 재조회 → 요약은 `last_run_summary.json`로 충분.

## 디버깅
- 이벤트 전건/리포트 전문을 stdout에 보려면 워크플로/로컬에서 `DEBUG=1`.
- 기본값(off)에서는 한 줄 요약만 출력되어 로그가 가볍다.
